# 第 1 章 绪论

## 1.1 研究背景与意义

### 1.1.1 社交媒体时代的舆情挑战

随着互联网基础设施的持续完善与智能移动终端的广泛普及，社交媒体已深度融入现代社会的信息传播体系之中。微博、抖音、小红书、B 站、知乎等平台日活跃用户规模均达亿级，用户每日产生的评论、转发与点赞行为构成了规模庞大、实时动态的舆论场域。根据中国互联网络信息中心（CNNIC）发布的第 53 次《中国互联网络发展状况统计报告》，截至 2023 年底，我国网民规模已超过 10.92 亿，网络视频用户规模达 10.67 亿，短视频平台的评论互动量更是持续走高。

在这一背景下，舆情信息呈现出三个典型特征：其一是**高速性**，热点事件往往在数分钟至数小时内完成从萌发到扩散的全过程；其二是**海量性**，单一话题在高峰时段内可产生百万量级的相关评论；其三是**多源异构性**，不同平台的内容结构、传播机制与用户群体存在显著差异，难以以统一范式进行处理。这三个特征共同构成了舆情监测领域的核心技术挑战。

### 1.1.2 传统舆情系统的局限性

传统舆情分析系统大多基于批处理（Batch Processing）架构，以定时任务的方式周期性地抓取数据、离线建模分析后输出报告。这一范式在应对低频率、小规模的历史数据分析时尚能胜任，但在实时舆情监测场景中暴露出明显的结构性缺陷：

- **时效性不足**：批处理周期通常以小时甚至天为单位，数据处理结果存在严重滞后，无法及时捕捉舆情演变的关键转折节点；
- **扩展性受限**：面对突发性流量峰值，传统系统缺乏弹性扩展机制，容易出现处理积压乃至系统崩溃；
- **告警响应迟缓**：依赖定时轮询的告警机制无法在事件爆发初期实现快速干预，可能错过舆情处置的最佳时间窗口；
- **存储与计算耦合度高**：传统方案通常将存储与计算紧耦合，历史数据的版本管理、回溯分析能力薄弱。

上述局限性在面对近年来频繁出现的突发舆情事件时尤为突出，迫切需要一套具备低延迟、高吞吐、可扩展能力的实时流式舆情分析解决方案。

### 1.1.3 研究意义

本课题的研究意义主要体现在以下几个维度：

**理论层面**，本研究探索了将大规模预训练语言模型（StructBERT）与分布式流计算框架（Spark Structured Streaming）相结合的工程实践路径，为流式 NLP 推理与实时数据湖写入的协同设计提供了参考范例。

**技术层面**，本系统实现了从数据采集、消息队列、流式处理、情感分析、热度建模到可视化展示的完整技术链路，验证了以 Kafka + Structured Streaming + Delta Lake + StructBERT 为核心的技术栈在实际舆情场景中的可行性与有效性。

**应用层面**，系统所提供的多话题实时监测、多粒度趋势分析与自动预警能力，可直接服务于政府应急管理部门的舆情研判、互联网企业的品牌声誉监控以及学术领域的社会舆论计算研究。

## 1.2 国内外研究现状

### 1.2.1 流式计算框架研究现状

在流式计算领域，业界主流框架经历了从第一代（Apache Storm）到第二代（Apache Flink、Spark Streaming）再到当前主流的结构化流处理（Structured Streaming）的演进历程。

Apache Storm 作为早期代表性实时计算框架，以拓扑（Topology）为计算模型，虽能实现真正的逐条处理，但其 at-least-once 语义保证及较高的编程复杂度限制了其大规模应用。Spark Streaming 基于微批次（Micro-Batch）模型，将流数据切分为固定时间窗口的 RDD 进行处理，在保持 Spark 生态兼容性的同时引入了若干延迟。Databricks 于 2016 年提出的 Structured Streaming 在 Spark 2.0 中正式引入，将流处理统一抽象为无界 DataFrame，支持事件时间（Event Time）语义、窗口聚合与水位线（Watermark）机制，同时通过 `foreachBatch` 接口提供了与批处理 API 的深度互操作能力，极大地简化了流批一体应用的开发复杂度 [5][6]。

Apache Flink 作为 Structured Streaming 的主要竞争方案，以其原生流处理模型（Native Streaming）、精确一次（Exactly-Once）语义以及更细粒度的状态管理能力而著称。然而，Flink 的运维复杂度相对较高，且与 Python 生态的结合程度（PyFlink）相较于 PySpark 仍有差距，这在实际工程选型中是不可忽视的成本因素。

### 1.2.2 情感分析技术研究现状

情感分析（Sentiment Analysis），又称意见挖掘（Opinion Mining），是自然语言处理（NLP）领域的核心研究方向之一，旨在从文本数据中自动识别并提取主观性信息。

早期研究方法以基于词典与规则的方法为主，通过构建情感词典（如 HowNet、知网、BosonNLP 情感词典）对文本进行极性判断，具有可解释性强、计算成本低的优点，但对语境依赖和新词覆盖能力有限。

随着深度学习的兴起，以 TextCNN、BiLSTM-Attention 为代表的神经网络方法在情感分析基准数据集上取得了显著进展。然而，此类方法对标注数据量依赖较高，跨领域泛化能力相对有限。

近年来，以 BERT [Devlin et al., 2019] 为代表的预训练语言模型（Pre-trained Language Model，PLM）彻底改变了 NLP 领域的研究范式。通过大规模无标注语料上的自监督预训练，再经少量标注数据的任务微调，PLM 在多项情感分析基准任务上实现了大幅度的性能突破。在中文情感分析领域，阿里巴巴达摩院提出的 StructBERT [Wang et al., 2020] 在 BERT 基础上引入了结构感知预训练目标，通过词序重建与句子结构预测任务增强了模型对文本结构信息的建模能力，在中文情感分类任务上展现出优异性能。刘勇等人的研究 [3] 表明，将主题与多元情感融合的网络舆情动态分析方法能够更全面地刻画舆论演变规律，为本文热度建模思路提供了参考依据。

### 1.2.3 数据湖仓技术研究现状

数据湖（Data Lake）概念由 James Dixon 于 2010 年提出，旨在以低成本、大规模存储原始异构数据。然而，传统数据湖（如基于 HDFS 的方案）面临数据质量难以保证、缺乏事务性保障、历史版本管理能力弱等问题，被业界称为"数据沼泽"困境。

湖仓一体（Lakehouse）架构 [Armbrust et al., 2021] 作为数据仓库与数据湖的融合方案应运而生。Delta Lake 作为该架构的开源实现，在对象存储（如阿里云 OSS、AWS S3）之上实现了 ACID 事务保证、数据版本管理（Time Travel）、Schema 演化支持以及高效的数据跳跃索引（Data Skipping），使得流批一体的数据写入与查询成为可能。胡登豪的研究 [1] 和李桂珍的研究 [2] 均指出，将流式计算结果实时写入具备事务性保障的数据湖是当前大数据处理领域的重要技术趋势，本文的存储层设计正是基于这一趋势展开的。

### 1.2.4 现有研究的不足与本文的切入点

综合以上分析，现有相关研究在以下方面仍存在一定不足：

（1）多数研究仅聚焦于单一技术环节（如情感分析算法或流计算框架），缺乏将多平台数据采集、实时流处理、NLP 推理与可视化展示整合为完整系统的工程化验证；

（2）舆情热度评估维度较为单一，多数方案仅以评论数量或转发数量作为热度代理指标，忽视了情感极性对热度传播的放大效应；

（3）现有系统对多话题并行监测与分钟级端到端延迟的兼顾能力有限。

本文针对上述不足，设计并实现了一套完整的端到端实时舆情分析系统，其技术方案的核心创新点在于：将 StructBERT 情感推理结果嵌入微批次处理循环，并以情感极性作为热度计算的权重因子之一，从而在流处理链路中实现了数据采集、语义理解与热度评估的统一计算。

## 1.3 研究内容与主要工作

本文的主要研究内容可概括为以下六个方面：

**（1）多平台社交媒体数据采集模块设计**

针对微博、B 站、抖音、小红书等主流社交媒体平台，设计并实现基于 Chrome DevTools Protocol（CDP）反检测技术的异步爬虫系统，支持关键词搜索与持续轮询两种采集模式，并将采集结果以 JSONL 格式写入 Kafka 生产者缓冲目录，构建数据入湖的源头。

**（2）Kafka 消息队列接入与流数据解析**

设计以 `public_opinion_raw` 为主题的 Kafka 消费管道，定义评论数据的标准化 JSON Schema，并在 Structured Streaming 层实现流数据的解析与字段校验，确保下游处理链路的数据质量。

**（3）StructBERT 情感分类服务设计**

以阿里达摩院 `iic/nlp_structbert_sentiment-classification_chinese-base` 模型为核心，设计单例化的批量推理服务，支持批量大小可配置（默认 32）、中性阈值动态调整（默认置信度 0.72）及异常降级处理，实现在 Spark Driver 端的高效情感推理。

**（4）多维热度评分模型与风险等级划定**

提出综合考量用户交互行为（点赞数、转发数）与情感极性权重的热度评分公式，并据此制定三级风险分类规则（高风险/中风险/低风险），为舆情态势的定量评估提供支撑。

**（5）Delta Lake 三层数据湖仓架构设计**

设计 Bronze-Silver-Gold 渐进式数据模型：Bronze 层追加写入 Kafka 原始消息与情感推理结果；Silver 层基于 `event_id` 字段进行 Merge 去重，保证数据一致性；Gold 层按话题与时间维度（小时/日）进行聚合统计，生成可供前端直接消费的分析指标。

**（6）FastAPI 服务层与 Vue 3 可视化大屏设计**

基于 FastAPI 框架封装 9 个 REST 接口，直接读取 Delta Lake 数据；前端采用 Vue 3 + ECharts 构建多话题切换、日/小时粒度趋势图、评论明细表与告警记录四大功能区域，结合玻璃拟态（Glassmorphism）设计风格，实现兼具美观性与可用性的监测大屏。

## 1.4 论文组织结构

本文共分为六章，各章节内容安排如下：

**第 1 章 绪论**：阐述研究背景与意义，综述国内外相关研究现状，明确研究内容与主要工作，介绍论文整体组织结构。

**第 2 章 相关技术与理论基础**：系统介绍本文所采用的核心技术，包括 Apache Kafka、Spark Structured Streaming、Delta Lake、StructBERT 情感分析模型、FastAPI 以及 Vue 3 + ECharts 等，为后续系统设计提供理论支撑。

**第 3 章 系统需求分析与总体设计**：从功能需求与非功能需求两个维度对系统进行需求分析，给出系统总体架构设计方案、数据流设计、模块划分以及关键接口定义。

**第 4 章 系统详细设计与实现**：按模块顺序详细阐述各功能子系统的设计思路、核心实现逻辑与关键代码分析，涵盖爬虫采集、Kafka 消费管道、StructBERT 推理服务、热度建模、Delta Lake 写入、告警机制、REST API 及前端可视化共八个子模块。

**第 5 章 系统测试与实验分析**：描述测试环境配置，从功能测试与性能测试两个维度对系统进行评估，分析实验结果并对系统局限性进行客观讨论。

**第 6 章 总结与展望**：对本文工作进行全面总结，指出当前系统存在的不足之处，并对未来可能的研究与改进方向进行展望。
