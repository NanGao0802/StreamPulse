# 第 3 章 系统需求分析与总体设计

## 3.1 功能需求分析

根据开题报告的任务书要求及实际舆情监测场景调研，本系统需满足以下六类核心功能需求。

### 3.1.1 多平台数据采集需求

系统需能够对接多个国内主流社交媒体平台，以关键词为维度对相关话题下的公开评论数据进行采集。具体要求包括：

- 支持微博（Weibo）、B 站（Bilibili）、抖音（Douyin）、小红书（XHS）、知乎（Zhihu）、快手（Kuaishou）、贴吧（Tieba）等不少于五个主流平台；
- 采集内容应包含评论正文、发布时间、点赞数量、转发数量等关键字段；
- 支持以轮询方式持续抓取最新内容，轮询间隔可配置；
- 采集结果以结构化 JSON 格式输出，兼容 Kafka 消息协议。

### 3.1.2 实时流数据接入需求

系统需构建稳定的流数据消费管道，满足以下要求：

- 以 Apache Kafka 为消息中间件，支持多生产者并发写入同一主题；
- Spark Structured Streaming 消费端能够在指定触发周期内完成对新增消息的处理；
- 消费进度需持久化存储，支持异常重启后的断点续传；
- 消息解析过程需对格式异常、字段缺失等情况进行容错处理，保证处理管道的健壮性。

### 3.1.3 情感分析与热度计算需求

系统需对每条评论进行情感极性判定与热度量化，具体要求包括：

- 情感分类需覆盖正向（positive）、负向（negative）、中性（neutral）三个类别；
- 情感推理过程需支持批量处理，以满足微批次场景下的吞吐要求；
- 热度评分需综合考量点赞数、转发数与情感极性三个维度；
- 风险等级需根据热度与情感标签的组合进行三级划定（高/中/低），为告警机制提供输入依据。

### 3.1.4 数据存储与管理需求

系统需提供可靠、可查询的数据持久化能力，具体要求包括：

- 原始数据、清洗数据与聚合指标应分层存储，层间数据质量递进；
- 存储层需支持 ACID 事务，防止并发写入导致的数据不一致；
- 聚合指标需覆盖小时粒度与日粒度两个时间维度；
- 支持话题维度的历史数据查询，数据范围至少覆盖近 365 天；
- 每个话题下需支持按热度排名的 Top10 评论记录存储。

### 3.1.5 负面舆情告警需求

系统需具备自动化舆情预警能力，具体要求包括：

- 当某话题在某统计小时内负面评论数量超过最低阈值（默认 10 条）且负面占比超过预设阈值（默认 40%）时，自动触发告警；
- 告警通知以电子邮件形式发送至指定收件人，邮件内容需包含话题名称、统计时段、评论数量、负面占比及高热度负面评论样例；
- 同一话题同一统计小时内重复满足触发条件时，不应重复发送告警（去重机制）；
- 告警事件需持久化存储，供后续查询与审计。

### 3.1.6 可视化展示需求

系统需提供直观、交互友好的数据展示界面，具体要求包括：

- 支持多话题切换，默认显示最新数据话题；
- 展示话题核心 KPI 指标（累计评论数、总热度、最新负面占比、告警次数）；
- 提供评论数、负面评论数、负面占比、热度四类趋势图表，支持按天和按小时两种粒度切换；
- 展示评论明细列表（含情感标签、情感分数、热度、原始文本等字段）；
- 展示历史告警记录及告警对应的负面评论样例；
- 图表支持缩放与拖拽交互，适配不同时间跨度的数据展示需求。

## 3.2 非功能需求分析

### 3.2.1 实时性需求

系统端到端处理延迟（从评论写入 Kafka 到前端可查询结果更新）需满足准实时要求，即在当前 60 秒微批次触发周期内，新增评论数据应在下一个完整批次处理完成后即可在前端呈现，整体端到端延迟控制在 120 秒以内。

### 3.2.2 可靠性需求

系统核心链路需具备容错恢复能力：Kafka 消费进度通过 Checkpoint 机制持久化，保证 Spark 任务异常重启后数据不丢失；Delta Lake 的 ACID 事务保证写入失败不会产生脏数据；告警去重机制保证不因重复处理产生告警轰炸。

### 3.2.3 可扩展性需求

系统架构需支持水平扩展：新增监测话题无需修改代码，仅需在爬虫端增加关键词配置即可自动纳入监测范围；新增社交媒体平台时，仅需在爬虫工厂（CrawlerFactory）注册新的爬虫类，不影响下游处理逻辑。

### 3.2.4 可维护性需求

各模块职责清晰，接口定义规范：爬虫模块、消息队列、流处理管道、API 服务与前端之间均通过标准化接口（Kafka JSON Schema、REST API）进行解耦；日志输出规范，关键处理节点有明确的进度与错误日志，便于问题定位。

### 3.3.2 技术架构关键决策

**决策一：微批次 vs. 纯流式**

本系统选择 Spark Structured Streaming 的微批次模式而非 Apache Flink 的原生流式处理。主要考量：其一，StructBERT 模型推理在 Driver 端以批量方式执行，微批次模式天然匹配批量推理的计算模式；其二，Delta Lake 的 Merge 操作（用于 Silver 层去重）在微批次粒度下效率更高，避免了逐条 Merge 的性能损耗；其三，PySpark 与 Python NLP 生态（ModelScope）的结合在成熟度上优于 PyFlink。

**决策二：`foreachBatch` vs. 标准 Sink**

标准 Structured Streaming Sink（如 `delta` 格式的 `writeStream`）仅支持单目标写入，而本系统需同时向 Bronze、Silver、多个 Gold 表及 Alert 表写入。`foreachBatch` 接口允许在每个批次内执行完整的批处理逻辑，完美支持多目标扇出（Fan-Out）写入场景。

**决策三：Delta Lake vs. 传统关系型数据库**

相较于将聚合结果写入 MySQL 或 PostgreSQL，Delta Lake 方案在存储成本（无服务器费用）、历史版本查询（Time Travel）以及流批一体（无需额外 ETL 管道）三个维度上具有显著优势，更适合实时分析场景。

## 3.4 数据流设计

### 3.4.1 数据采集阶段数据流

爬虫进程以关键词为单位，通过 Playwright CDP 自动化浏览器抓取各平台公开评论页面，从 HTML/JSON 响应中提取评论元数据，经字段归一化处理后序列化为如下 JSON Schema 的消息格式，并写入本地 JSONL 文件目录，由独立的 Kafka Producer 进程消费该目录并推送至 Kafka 主题：

```json
{
  "event_id":     "<string>  唯一事件标识，用于全系统去重",
  "platform":     "<string>  来源平台标识（bili/wb/xhs/dy/ks/tieba/zhihu）",
  "topic":        "<string>  话题关键词",
  "publish_time": "<string>  评论发布时间（yyyy-MM-dd HH:mm:ss）",
  "time_id":      "<string>  时间桶标识",
  "text":         "<string>  评论正文",
  "like_count":   "<long>    点赞数",
  "repost_count": "<long>    转发数",
  "ingest_time":  "<string>  数据入库时间戳",
  "source_file":  "<string>  来源文件路径（用于溯源）"
}
```

### 3.4.2 流处理阶段数据流

Spark Structured Streaming 从 Kafka 读取消息后，经以下处理步骤生成各层数据：

1. **Kafka 元数据提取**：提取 `key`、`value`、`topic`、`partition`、`offset`、`timestamp` 六个 Kafka 元数据字段；
2. **JSON 解析与字段过滤**：通过 `from_json` 函数按预定义 Schema 解析消息体，过滤掉 `event_id`、`text`、`publish_time`、`topic` 任一为空的记录；
3. **StructBERT 情感推理**：批量调用情感分类 Pipeline，为每条记录追加 `sentiment_label`、`sentiment_score`、`sentiment_confidence`、`sentiment_source`、`sentiment_model` 五个情感字段；
4. **衍生字段计算**：计算 `publish_ts`（时间戳类型）、`text_len`（文本长度）、`heat_score`（热度分数）、`risk_level`（风险等级）、`event_hour`（小时桶）、`event_date`（日期桶）；
5. **多层写入**：按照 Bronze → Silver → Gold → Alert 顺序执行写入，各层写入逻辑独立，互不影响。

### 3.4.3 数据服务阶段数据流

FastAPI 服务收到前端 HTTP 请求后，通过 `deltalake` 库直接读取 Delta Lake 表文件，转换为 Pandas DataFrame 进行过滤、排序与序列化处理，最终以 JSON 格式返回。整个读取过程无需启动 SparkSession，避免了 JVM 启动开销，显著降低了 API 响应延迟。

## 3.5 模块划分

根据系统架构设计，将系统划分为以下八个功能子模块：

| 编号 | 模块名称 | 对应文件/目录 | 主要职责 |
|------|----------|--------------|---------|
| M1 | 多平台爬虫模块 | `crawler/` | 多平台评论采集、JSONL 输出 |
| M2 | Kafka 消费与解析模块 | `jobs/stream_single_topic_to_delta.py` → `main()` | Kafka 接入、Schema 解析、字段过滤 |
| M3 | 情感分析服务模块 | `jobs/structbert_sentiment.py` | StructBERT 批量推理、标签归一化、异常降级 |
| M4 | 热度计算与风险评估模块 | `jobs/stream_single_topic_to_delta.py` → `build_enriched_df()` | 热度公式计算、风险等级划定 |
| M5 | Delta Lake 数据湖写入模块 | `jobs/stream_single_topic_to_delta.py` → `process_batch()` / `rebuild_gold_tables()` | Bronze/Silver/Gold 三层写入 |
| M6 | 告警检测与邮件通知模块 | `jobs/alert_mailer.py`、`check_and_send_alerts()` | 阈值检测、邮件发送、告警持久化 |
| M7 | REST API 数据服务模块 | `api/app.py` | 9 个端点的数据查询与序列化 |
| M8 | 前端可视化模块 | `frontend/` | 大屏渲染、交互逻辑、图表更新 |

## 3.6 关键接口定义

### 3.6.1 Kafka 消息接口

- **Topic 名称**：`public_opinion_raw`
- **消息格式**：UTF-8 编码的 JSON 字符串
- **必填字段**：`event_id`、`topic`、`publish_time`、`text`
- **可选字段**：`like_count`（缺省为 0）、`repost_count`（缺省为 0）、`platform`、`ingest_time`、`source_file`

### 3.6.2 REST API 接口规范

所有 API 端点均为 HTTP GET 方法，响应格式为 JSON。核心端点规范如下：

| 端点 | 参数 | 响应字段 |
|------|------|---------|
| `GET /api/dashboard` | `topic`、`daily_limit`、`hourly_limit`、`alerts_limit` | `meta`、`summary`、`daily`、`hourly`、`alerts` |
| `GET /api/comments` | `topic`、`view`（daily/hourly）、`date`、`hour`、`limit` | `topic`、`view`、`selected_bucket`、`bucket_options`、`rows`、`data` |
| `GET /api/alerts` | `topic`、`limit` | `topic`、`rows`、`data` |

### 3.6.3 告警触发规则接口

告警触发条件通过环境变量进行配置，以实现无需修改代码的参数调整：

| 环境变量名 | 默认值 | 含义 |
|-----------|--------|------|
| `ALERT_MIN_COMMENTS` | `10` | 触发告警的最低评论数阈值 |
| `ALERT_NEG_RATIO_THRESHOLD` | `0.40` | 触发告警的负面评论占比阈值 |
| `ALERT_TOPN_TEXTS` | `3` | 告警邮件中附带的高热度负面评论样例数 |
| `STRUCTBERT_NEUTRAL_THRESHOLD` | `0.72` | 中性标签判定的置信度阈值 |
| `STRUCTBERT_BATCH_SIZE` | `32` | 情感推理的批量大小 |
