# 第 4 章 系统详细设计与实现

## 4.1 多平台社交媒体数据采集模块

### 4.1.1 模块设计目标

数据采集模块的核心目标是在合规、稳定、低干扰的前提下，以关键词为维度对多个主流社交媒体平台的公开评论数据进行周期性采集，并将结果以标准化 JSON 格式输出，供 Kafka 生产者消费。

### 4.1.2 爬虫工厂模式

本模块采用工厂模式（Factory Pattern）对多平台爬虫进行统一管理。`CrawlerFactory` 类维护一个平台标识到爬虫类的映射字典，通过 `create_crawler(platform)` 静态方法按需实例化对应的爬虫对象，各平台爬虫类均实现统一的 `AbstractCrawler` 抽象接口，保证了下游调用逻辑的一致性：

```python
class CrawlerFactory:
    CRAWLERS = {
        "xhs":   XiaoHongShuCrawler,
        "dy":    DouYinCrawler,
        "ks":    KuaishouCrawler,
        "bili":  BilibiliCrawler,
        "wb":    WeiboCrawler,
        "tieba": TieBaCrawler,
        "zhihu": ZhihuCrawler,
    }

    @staticmethod
    def create_crawler(platform: str) -> AbstractCrawler:
        crawler_class = CrawlerFactory.CRAWLERS.get(platform)
        if not crawler_class:
            raise ValueError(f"不支持的平台: {platform}")
        return crawler_class()
```

### 4.1.3 CDP 反检测机制

传统 Playwright 自动化采用无头浏览器（Headless Browser）模式，各主流平台已对此实施了较为成熟的检测机制（如 `navigator.webdriver` 属性检测、Canvas 指纹检测等）。本系统配置 `ENABLE_CDP_MODE = True`，通过 Chrome DevTools Protocol（CDP）接管用户本地已安装的 Chrome/Edge 实例。这一模式的优势在于：

- 复用用户真实浏览器的 Cookie、扩展及用户 Agent，降低被识别为自动化程序的风险；
- 真实浏览器的 TLS 指纹与 Canvas 指纹与正常用户行为完全一致；
- 支持手动介入处理滑动验证码等人机验证环节，再由程序接管后续操作。

CDP 调试端口默认为 9222，系统在启动时自动检测 Chrome/Edge 安装路径（`CUSTOM_BROWSER_PATH` 可手动指定），并通过 Playwright 的 `connect_over_cdp` API 建立控制连接。

### 4.1.4 轮询采集模式

为满足实时性要求，本模块支持轮询（Polling）采集模式（`ENABLE_POLL_LATEST = True`）：程序在完成一轮关键词搜索后，等待 `POLL_INTERVAL_SEC` 秒（默认 30 秒，可通过环境变量 `CRAWL_INTERVAL` 覆盖），随后再次执行搜索，形成持续采集循环。每轮采集获取最新的 `SEARCH_PAGES` 页（默认 2 页）评论，以捕捉新增内容。

### 4.1.5 JSONL 输出设计

爬取到的评论经字段归一化处理后，以每行一个 JSON 对象的 JSONL（JSON Lines）格式追加写入 `KAFKA_JSON_DIR` 指定目录，文件命名格式为 `{platform}_{topic}_{date}.jsonl`。JSONL 格式的优势在于支持流式读取，Kafka 生产者进程可通过 `tail -f` 等方式实时监听新增行并推送消息，实现爬虫与 Kafka 之间的松耦合数据交换。

### 4.1.6 数据存储扩展能力

模块支持多种数据存储后端（由 `SAVE_DATA_OPTION` 配置）：`postgresql` 模式将数据持久化至关系型数据库，具备字段约束与排重能力；`csv` 模式生成标准 CSV 文件；`json` 模式输出 JSONL；`sqlite` 模式适用于本地轻量化调试。同时，模块内置了 CSV 转 XLSX 聚合功能（`export_csvs_to_xlsx`），支持将同一次采集会话中多个 CSV 文件合并为多 Sheet 的 Excel 工作簿，便于人工审阅。

## 4.2 Kafka 消费管道与数据解析模块

### 4.2.1 Kafka 读取配置

本系统通过 Structured Streaming 的 Kafka Source 接入消息流，关键配置如下：

```python
kafka_df = (
    spark.readStream
    .format("kafka")
    .option("kafka.bootstrap.servers", "172.16.156.151:9092")
    .option("subscribe", "public_opinion_raw")
    .option("startingOffsets", "latest")      # 仅消费新增消息
    .option("failOnDataLoss", "false")         # 丢失偏移量时不中断任务
    .option("maxOffsetsPerTrigger", "50")      # 每次微批最多处理 50 条
    .load()
)
```

`startingOffsets = "latest"` 的设置意味着系统启动时从当前最新偏移量开始消费，不处理历史积压消息，确保监测结果反映当前实时舆情状态。`failOnDataLoss = "false"` 则防止 Kafka 日志过期删除导致任务失败。

### 4.2.2 JSON 解析与字段校验

Kafka 消息的 `value` 字段为 UTF-8 编码的 JSON 字符串。系统预定义 `StructType` Schema，通过 Spark 内置的 `from_json` 函数进行结构化解析，并追加 Kafka 元数据字段（`kafka_topic`、`kafka_partition`、`kafka_offset`、`kafka_timestamp`）以实现完整的消息溯源能力。

解析完成后，对以下四个关键字段进行非空过滤，确保不合规消息不进入下游处理链路：

```python
.filter(F.col("event_id").isNotNull())
.filter(F.col("text").isNotNull())
.filter(F.col("publish_time").isNotNull())
.filter(F.col("topic").isNotNull())
.filter(F.length(F.col("topic")) > 0)
```

同时对 `topic` 字段执行 `trim()` 操作去除首尾空白，防止因编码差异导致的话题分组错误。

## 4.3 StructBERT 情感分析服务模块

### 4.3.1 单例服务设计

模型加载是推理服务中最耗时的操作（通常需要数秒至数十秒）。为避免每次微批次处理时重复加载权重，`StructBERTSentimentService` 采用懒初始化单例模式（Lazy Singleton）：

```python
class StructBERTSentimentService:
    _instance = None

    @classmethod
    def get_instance(cls):
        if cls._instance is None:
            cls._instance = StructBERTSentimentService()
        return cls._instance
```

类变量 `_instance` 在首次调用 `get_instance()` 时被初始化，此后所有批次复用同一模型实例，模型权重常驻 Driver 进程内存，实现了一次加载、多次复用的高效推理模式。

### 4.3.2 批量推理流程

`predict_texts` 方法接收文本列表，按 `batch_size`（默认 32）进行分组，依次提交给 ModelScope Pipeline 执行推理：

```python
def predict_texts(self, texts: List[Any]) -> List[Dict[str, Any]]:
    clean_texts = [self._normalize_text(t) for t in texts]
    results = []
    for i in range(0, len(clean_texts), self.batch_size):
        chunk = clean_texts[i:i + self.batch_size]
        chunk_result = self._predict_chunk(chunk)
        results.extend(chunk_result)
    return results
```

文本预处理（`_normalize_text`）包含以下操作：去除首尾空白、将换行符替换为空格、截断超过 `max_text_length`（默认 256）个字符的文本，以确保输入符合模型上下文窗口限制。

### 4.3.3 标签归一化与中性判定

ModelScope Pipeline 输出的原始标签形式存在 `label_0`/`label_1` 或 `positive`/`negative` 等多种变体，`_canonical_label` 方法将其统一规范化：

```python
def _canonical_label(self, label: Any) -> str:
    s = str(label).strip().lower()
    if s in {"1", "label_1", "positive", "pos"}:
        return "positive"
    if s in {"0", "label_0", "negative", "neg"}:
        return "negative"
    return "unknown"
```

在最终标签决策时，引入置信度阈值进行中性判定：当模型对预测标签的置信度低于 `neutral_threshold`（默认 0.72）时，认为该评论情感倾向不明确，将其归为"中性"类别，情感分数置为 0.0，从而降低低置信度预测对热度统计的干扰：

```python
if confidence < self.neutral_threshold:
    final_label = "neutral"
else:
    final_label = raw_label if raw_label in {"positive", "negative"} else "neutral"
```

### 4.3.4 异常降级策略

批量推理失败时，系统自动退化为逐条推理模式；若单条推理仍然失败，则以 `neutral/0.0` 作为兜底结果返回，保证流处理管道不因推理异常而中断：

```python
except Exception as e:
    outputs.append({
        "sentiment_label": "neutral",
        "sentiment_confidence": 0.0,
        "sentiment_score": 0.0,
        "sentiment_source": "structbert",
        ...
    })
```

## 4.4 热度计算与风险评估模块

### 4.4.1 热度评分模型

本系统提出的热度评分（Heat Score）模型综合考量三个维度：**用户点赞行为**（反映内容正向传播意愿）、**用户转发行为**（反映内容扩散意愿，权重高于点赞）以及**情感极性权重**（反映负面情绪的舆论放大效应）。公式定义如下：

$$\text{heat\_score} = 1.0 + \text{like\_count} \times 0.6 + \text{repost\_count} \times 1.0 + \text{sentiment\_bonus}$$

其中，情感极性权重 $\text{sentiment\_bonus}$ 的取值规则为：

$$\text{sentiment\_bonus} = \begin{cases} 1.5 & \text{若 sentiment\_label} = \text{negative} \\ 0.5 & \text{若 sentiment\_label} = \text{neutral} \\ 0.2 & \text{若 sentiment\_label} = \text{positive} \end{cases}$$

设计依据分析：其一，转发行为的社会传播效应强于点赞，故转发权重（1.0）高于点赞权重（0.6）；其二，负面评论在社交媒体中具有更强的情绪感染力和二次传播动力，故其权重加成最高（1.5），这与刘勇等人 [3] 关于负面舆情放大效应的研究结论相一致；其三，基础分 1.0 保证无互动数据的纯文本评论亦有基本热度值，避免出现零值。

在 Spark DataFrame 层面，热度计算通过以下表达式实现：

```python
.withColumn(
    "heat_score",
    F.round(
        F.lit(1.0)
        + F.col("like_count") * F.lit(0.6)
        + F.col("repost_count") * F.lit(1.0)
        + F.when(F.col("sentiment_label") == "negative", F.lit(1.5))
           .when(F.col("sentiment_label") == "neutral", F.lit(0.5))
           .otherwise(F.lit(0.2)),
        2
    )
)
```

### 4.4.2 风险等级划定规则

基于情感标签与热度分数的组合，系统对每条评论划定三级风险等级，为告警决策提供精细化依据：

| 条件 | 风险等级 | 含义 |
|------|---------|------|
| `sentiment_label == "negative"` 且 `heat_score ≥ 12` | `high` | 高热度负面评论，需即刻关注 |
| `sentiment_label == "negative"` 且 `heat_score ≥ 5` | `medium` | 中等热度负面评论，需持续监测 |
| 其他情况 | `low` | 低风险，常规记录 |

热度阈值 12 的设定依据：点赞数约为 18（18×0.6=10.8）或转发数约为 11（11×1.0=11.0）即可在无额外互动的情况下达到该阈值，对应社交媒体上具有一定传播规模的评论。

## 4.5 Delta Lake 三层数据湖写入模块

### 4.5.1 Bronze 层：原始数据落盘

Bronze 层以追加（Append）模式写入，保存 Kafka 消息的完整信息（含 Kafka 元数据字段）及情感推理结果。写入按 `event_date` 字段进行分区，以优化基于日期的历史查询性能：

```python
bronze_df.write \
    .format("delta") \
    .mode("append") \
    .partitionBy("event_date") \
    .save(BRONZE_PATH)
```

Bronze 层不做任何业务逻辑过滤，其核心价值在于保留数据的完整历史记录，是系统数据的最终真相（Ground Truth），支持上层数据出错时的回溯重建。

### 4.5.2 Silver 层：去重精炼写入

Silver 层基于 `event_id` 字段进行 Merge 去重，确保同一条评论不被重复计算。当 Silver 表不存在时，首次写入使用 Append 模式初始化；此后采用 Delta Lake 的 `merge` 操作，仅插入不存在的新记录：

```python
target.alias("t") \
    .merge(
        dedup_df.alias("s"),
        "t.event_id = s.event_id"
    ) \
    .whenNotMatchedInsertAll() \
    .execute()
```

Merge 操作利用 Delta Lake 的事务日志实现原子性，即使并发写入也不会产生重复数据。Silver 层剔除了 Kafka 元数据字段，仅保留业务相关字段，数据模型更加精炼。

### 4.5.3 Gold 层：多粒度聚合指标

Gold 层在每次批次处理完成后，基于最新 Silver 数据进行全量重算（`overwriteSchema=true` 的 Overwrite 模式），生成三类聚合表：

**小时聚合表（Gold Hourly）**：以 `(topic, stat_hour)` 为分组键，统计评论数、负面评论数、负面占比、热度总和与热度均值：

```python
hourly_df = (
    silver_df
    .groupBy(
        F.col("topic"),
        F.date_trunc("hour", F.col("publish_ts")).alias("stat_hour")
    )
    .agg(
        F.count("*").alias("comment_count"),
        F.sum(F.when(F.col("sentiment_label") == "negative", 1)
               .otherwise(0)).alias("negative_count"),
        F.round(F.sum("heat_score"), 2).alias("heat_sum"),
        F.round(F.avg("heat_score"), 2).alias("heat_avg"),
    )
    .withColumn(
        "negative_ratio",
        F.round(
            F.when(F.col("comment_count") > 0,
                   F.col("negative_count") / F.col("comment_count"))
             .otherwise(F.lit(0.0)), 4
        )
    )
)
```

**日聚合表（Gold Daily）**：以 `(topic, stat_date)` 为分组键，统计口径与小时聚合表一致，粒度提升至天维度。

**Top10 评论表（Gold Top10）**：在 Silver 数据中，按 `(topic, event_date)` 分组，依据 `heat_score`（主序）、`like_count`（次序）、`repost_count`（三序）进行窗口排名，提取每组前 10 名记录，用于前端"最热评论"展示功能。

### 4.5.4 数据一致性保障

Silver 层的 Merge 操作与 Gold 层的 Overwrite 操作均在 `foreachBatch` 函数体内顺序执行。由于 Spark Driver 以单线程方式串行执行每个微批次的 `foreachBatch` 回调，天然避免了并发写入同一 Delta 表的冲突；Delta Lake 的事务日志则在节点重启或 I/O 异常时提供原子性保障。

## 4.6 负面舆情告警模块

### 4.6.1 告警触发逻辑

告警检测在每次微批次处理的 Gold 层重算完成后执行，以确保基于最新聚合结果进行判断。核心逻辑如下：

1. 从当前批次的 enriched DataFrame 中提取涉及的 `(topic, event_hour)` 组合；
2. 关联 Gold Hourly 表，筛选满足双重阈值条件的记录：`comment_count > ALERT_MIN_COMMENTS` 且 `negative_ratio > ALERT_NEG_RATIO_THRESHOLD`；
3. 查询 Alert 表中已存在的告警记录键集合（`alert_key = f"{topic}__{stat_hour}"`），实现基于历史记录的去重；
4. 对未重复的候选告警，从 Silver 层查询该话题该小时内热度最高的 `ALERT_TOPN_TEXTS` 条负面评论作为样例；
5. 调用 `AlertMailer` 发送告警邮件，并将告警事件记录写入 Alert Delta 表。

### 4.6.2 告警邮件设计

告警邮件通过 Python 标准库 `smtplib` 发送，支持 SSL 与 STARTTLS 两种加密模式（默认 SSL，端口 465），兼容 QQ 邮箱、163 邮箱等国内主流 SMTP 服务。邮件正文格式如下：

```
检测到舆情预警，请及时关注：

话题：{topic}
统计小时：{stat_hour}
该小时评论总数：{comment_count}
该小时负面评论数：{negative_count}
该小时负面占比：{negative_ratio:.2%}

该话题该小时内热度最高的负面 3 条评论如下：
1. 发布时间：{publish_time}
   热度：{heat_score}
   评论：{text}
...

该邮件由实时舆情分析与预警系统自动发送。
```

### 4.6.3 告警配置管理

所有告警相关配置通过 `/opt/pipeline/conf/alert.env` 环境变量文件进行管理，实现配置与代码的分离：

```bash
ALERT_EMAIL_ENABLED=true
SMTP_HOST=smtp.qq.com
SMTP_PORT=465
SMTP_USE_SSL=true
SMTP_USER=your_email@qq.com
SMTP_PASSWORD=your_auth_code
SMTP_FROM=your_email@qq.com
SMTP_TO=receiver@example.com
ALERT_NEG_RATIO_THRESHOLD=0.40
ALERT_MIN_COMMENTS=10
ALERT_TOPN_TEXTS=3
```

## 4.7 REST API 数据服务模块

### 4.7.1 数据读取策略

API 服务通过 `deltalake` Python 库（Delta Lake 的纯 Python 实现，无需 JVM）直接读取 Delta Lake 目录，转换为 Pandas DataFrame 后进行内存级的过滤、排序与序列化处理。相较于通过 SparkSession 读取的方案，这一策略显著降低了每次 API 请求的响应延迟（无 JVM 启动开销）和资源消耗（无 Spark Executor）。

### 4.7.2 数据序列化处理

Pandas DataFrame 在转换为 JSON 过程中存在若干类型兼容性问题，如 `pd.Timestamp` 无法直接序列化为 JSON 字符串、NumPy 数值类型（`np.int64`、`np.float64`）不兼容标准 JSON 等。本系统通过 `normalize_value` 函数对每个字段值进行类型归一化处理：

```python
def normalize_value(v: Any) -> Any:
    if isinstance(v, pd.Timestamp):
        return v.strftime("%Y-%m-%d %H:%M:%S")
    if hasattr(v, "item"):      # NumPy 标量
        return v.item()
    if isinstance(v, float) and math.isnan(v):
        return None
    return v
```

### 4.7.3 核心接口实现

**`/api/dashboard` 聚合接口**：一次性返回 meta、summary、daily、hourly、alerts 五组数据，减少前端在页面加载时的 HTTP 请求次数，提升首屏渲染速度。

**`/api/comments` 评论明细接口**：支持 `view=daily`（按天，返回热度前 30 条）和 `view=hourly`（按小时，返回最多 2000 条）两种模式，并在响应中附带 `bucket_options`（可选时间桶列表）与 `selected_bucket`（当前选中时间桶），使前端下拉选择框无需额外请求即可渲染。

**`/api/topics` 话题列表接口**：从 Gold Daily 表中聚合所有历史话题，按评论总量降序排列，使最活跃话题排列在前，符合用户优先关注热点话题的使用习惯。

## 4.8 前端可视化模块

### 4.8.1 整体布局设计

前端采用单页应用（SPA）架构，全部逻辑集中于 `App.vue` 单文件组件。页面布局分为四个垂直区域：

1. **Hero 头部区域**：左侧展示系统标题与技术标签徽章，右侧为控制面板（话题选择、粒度切换、时间桶选择、刷新按钮）；
2. **KPI 指标卡区域**：六个等宽指标卡，展示核心汇总数据；
3. **趋势图表区域**：三张 ECharts 折线图（2+1 布局，热度图全宽），支持 DataZoom；
4. **明细内容区域**：左侧评论明细表（占 1.55 份宽度）与右侧告警记录面板（占 1 份宽度）并排展示。

页面整体采用玻璃拟态（Glassmorphism）设计语言，通过 `backdrop-filter: blur(18px)` 与半透明背景色营造现代化的视觉层次感，符合当前数据大屏的主流审美趋势。

### 4.8.2 数据获取与状态管理

页面采用 Vue 3 Composition API 进行状态管理，核心响应式数据包括：

- `selectedTopic`：当前选中话题（字符串）
- `granularity`：分析粒度（`"daily"` 或 `"hourly"`）
- `dashboard`：包含 meta、summary、daily、hourly、alerts 的汇总数据对象
- `comments`：评论明细数据对象（含 `bucket_options`、`data`）
- `alerts`：告警记录数据对象

数据获取逻辑集中于 `refreshAll` 函数，在以下场景中触发：页面初始化（`onMounted`）、点击刷新按钮、切换话题（`handleTopicChange`）。粒度切换（`switchGranularity`）仅重新请求评论数据，不重置 dashboard 数据，避免不必要的网络请求。

### 4.8.3 ECharts 图表渲染

三个 ECharts 实例分别挂载于 `countChartRef`、`ratioChartRef`、`heatChartRef` 三个 DOM 引用上，通过 `renderCharts` 函数统一更新：

```javascript
function renderCharts() {
    if (!countChart) countChart = echarts.init(countChartRef.value);
    countChart.setOption(countOption(), true);
    // 同理处理 ratioChart 和 heatChart
}
```

`watch(trendRows, ...)` 深度监听趋势数据变化，在数据更新后自动触发图表重渲染，保证图表内容与数据状态的实时同步。`window.resize` 事件监听确保浏览器窗口调整时图表自适应重绘（`resizeCharts`）。

所有图表的 X 轴均使用计算属性 `xLabels` 统一生成，按粒度不同分别截取日期（`yyyy-MM-dd`）或小时（`MM-dd HH:mm`）格式，并通过 `axisLabel.rotate: 30` 倾斜显示，防止标签重叠。

### 4.8.4 响应式适配

页面通过 CSS 媒体查询实现多分辨率适配：

- **≥ 1600px**：KPI 卡 6 列展示，趋势图 2 列布局，评论表与告警面板并排；
- **1200px ～ 1600px**：KPI 卡切换为 3 列；
- **< 1200px**：Hero 区域、内容区域、图表区域切换为单列纵向堆叠；
- **< 700px**：KPI 卡切换为单列，标题字号缩小，页面边距压缩。

## 4.9 系统部署配置

### 4.9.1 云服务器端部署

系统后端部署于阿里云 Linux 3（等效 Ubuntu 22.04 LTS）服务器，各组件路径约定如下：

| 组件 | 部署路径 | 启动方式 |
|------|---------|---------|
| Spark Streaming 任务 | `/opt/pipeline/jobs/` | `bash scripts/run_stream_single_topic.sh` |
| FastAPI 服务 | `/opt/pipeline/api/` | `bash scripts/run_api.sh` |
| 配置文件 | `/opt/pipeline/conf/alert.env` | 环境变量注入 |
| Delta Lake 数据目录 | `/data/public-opinion/delta/` | 自动创建 |
| Checkpoint 目录 | `/data/public-opinion/checkpoints/` | 自动创建 |

### 4.9.2 Spark 提交配置

Spark 任务通过 `spark-submit` 命令提交，关键参数如下：

```bash
spark-submit \
  --master local[*] \
  --packages io.delta:delta-spark_2.12:3.3.2,\
             org.apache.spark:spark-sql-kafka-0-10_2.12:3.5.8 \
  --conf spark.driver.memory=2g \
  --conf spark.executor.memory=2g \
  --conf spark.sql.session.timeZone=Asia/Shanghai \
  /opt/pipeline/jobs/stream_single_topic_to_delta.py
```

`local[*]` 模式指定使用本机全部 CPU 核心，适合单机部署场景；`spark.sql.session.timeZone=Asia/Shanghai` 确保时间截断（`date_trunc`）以中国标准时间为基准，防止小时桶计算出现时区偏移。
