# 第 2 章 相关技术与理论基础

## 2.1 Apache Kafka 消息队列

### 2.1.1 Kafka 基本架构

Apache Kafka 是由 LinkedIn 公司开发并于 2011 年开源的分布式流式消息平台，现已成为大数据实时管道领域事实上的工业标准。Kafka 采用发布-订阅（Publish-Subscribe）消息模型，其核心架构由以下组件构成：

- **Producer（生产者）**：负责将消息发布到指定 Topic；
- **Broker（代理节点）**：Kafka 集群的服务节点，负责消息的持久化存储与路由转发；
- **Topic（主题）**：消息的逻辑分类单元，每个 Topic 可划分为若干 Partition（分区）；
- **Partition（分区）**：Topic 的物理存储单元，消息在分区内以追加日志（Append-Only Log）形式写入，每条消息赋予单调递增的 Offset；
- **Consumer Group（消费者组）**：一组协同消费同一 Topic 的消费者实例，每个分区同一时刻只被组内一个消费者消费，实现并行处理与负载均衡；
- **ZooKeeper / KRaft**：早期版本通过 ZooKeeper 管理集群元数据，2.8 版本起引入 KRaft 模式以消除对 ZooKeeper 的依赖。

### 2.1.2 Kafka 的核心特性

**高吞吐量**：Kafka 通过顺序磁盘 I/O、零拷贝（Zero-Copy）传输、批量发送与压缩等机制，在普通硬件上可实现单节点每秒百万级消息的写入与读取能力，满足高并发场景下的数据接入需求。

**消息持久化与重放能力**：消息以追加日志形式持久化存储于磁盘，支持基于时间或 Offset 的历史消息重放，这一特性使得消费者在异常恢复后能够从断点处继续处理，有效保证了数据的不丢失性。

**精确偏移量控制**：Structured Streaming 通过记录已消费的 Kafka Offset 并存储于 Checkpoint 目录，配合 Kafka 的偏移量管理机制，在系统重启后能够从正确位置继续消费，实现端到端的 At-Least-Once 甚至 Exactly-Once 语义保证。

### 2.1.3 本系统中 Kafka 的应用

本系统中，Kafka 承担爬虫层与 Spark 流处理层之间的数据缓冲职责。爬虫进程将采集到的社交媒体评论序列化为 JSON 字符串后写入名为 `public_opinion_raw` 的主题，Broker 地址为 `172.16.156.151:9092`。Structured Streaming 作为消费者，以 `latest` 为起始偏移量、每次最多消费 50 条消息（`maxOffsetsPerTrigger=50`）的方式订阅该主题，从而实现生产者与消费者的解耦及流量削峰。

## 2.2 Apache Spark Structured Streaming

### 2.2.1 Structured Streaming 的设计哲学

Structured Streaming 是 Apache Spark 从 2.0 版本起引入的高层流处理 API，其核心设计哲学是将**流处理统一抽象为对无界表（Unbounded Table）的持续查询**。在该模型下，流数据被视为追加到无界表中的新行，每次触发（Trigger）时 Spark 对新增数据执行查询计划，将结果写入输出端（Sink）。这一模型使得开发者能够复用 Spark SQL 与 DataFrame API，以接近批处理的编程体验开发流处理应用，显著降低了开发门槛。

### 2.2.2 微批次处理模型

Structured Streaming 默认采用微批次（Micro-Batch）处理模式，即以固定时间间隔（Processing Time Trigger）为周期，将时间窗口内累积的流数据作为一个批次（Batch）进行处理。本系统将触发周期设置为 60 秒（`processingTime="60 seconds"`），在该周期内：

1. Spark 从 Kafka 读取最新批次的消息（不超过 `maxOffsetsPerTrigger` 条）；
2. 执行 JSON 解析与字段过滤；
3. 调用 `foreachBatch` 回调函数，在 Driver 端执行情感推理与 Delta Lake 写入操作；
4. 更新 Checkpoint 目录中的消费位点记录。

### 2.2.3 `foreachBatch` 接口

`foreachBatch` 是 Structured Streaming 提供的一个关键接口，允许用户对每个微批次的 DataFrame 执行任意批处理逻辑，包括非幂等写操作、复杂的多目标写入（Fan-Out）以及与外部系统的交互。本系统中，情感推理（通过 Python ModelScope Pipeline 调用）、Delta Lake 的多层写入（Bronze/Silver/Gold）以及告警检测逻辑均封装在 `foreachBatch` 回调函数 `process_batch` 中，实现了计算逻辑的高度内聚。

### 2.2.4 Checkpoint 与容错保障

Structured Streaming 通过 Checkpoint 机制实现故障恢复：每次微批次处理完成后，引擎自动将当前的消费偏移量、查询进度等状态信息序列化写入 HDFS 或本地文件系统中的 Checkpoint 目录。当 Spark 应用因异常崩溃重启后，引擎读取 Checkpoint 状态，从上次成功处理的断点处重新开始，保证消息不重不漏。本系统的 Checkpoint 路径设置为 `/data/public-opinion/checkpoints/single_topic_comment_pipeline`。

## 2.3 Delta Lake 数据湖仓架构

### 2.3.1 Delta Lake 概述

Delta Lake 是由 Databricks 于 2019 年开源的数据湖存储层框架，现已捐献给 Linux Foundation 并成为 Delta Lake 开源项目。其核心在于在普通对象存储（如 HDFS、S3、阿里云 OSS）之上，通过一个额外的事务日志层（Transaction Log，即 `_delta_log` 目录）实现了关系型数据库级别的 ACID 事务保证。

### 2.3.2 核心特性

**ACID 事务**：Delta Lake 的事务日志采用乐观并发控制（Optimistic Concurrency Control），每次写操作生成一个原子提交（Commit），确保并发读写场景下的数据一致性。

**数据版本管理（Time Travel）**：每次写入操作均在事务日志中记录版本号（Version），用户可通过指定版本号或时间戳查询历史快照，支持数据审计、回溯分析与增量对比。

**Schema 强制与演化**：Delta Lake 在写入时强制校验数据 Schema，防止脏数据污染；同时支持 Schema 演化（Schema Evolution），允许在不破坏已有数据的前提下新增列。

**高效读取优化**：通过数据跳跃（Data Skipping）和列统计信息（Column Statistics），Delta Lake 可在查询时跳过不相关的数据文件，显著降低 I/O 开销，尤其对于带有分区过滤条件的查询效果显著。

**流批一体写入**：Delta Lake 同时支持 Structured Streaming 的流式写入（Append/Merge）与 Spark DataFrame 的批量写入（Overwrite），使同一物理存储层能够同时承载实时数据管道与批量分析任务。

### 2.3.3 三层数据架构设计思想（Bronze-Silver-Gold）

Medallion Architecture（奖章架构）是 Databricks 提出的数据湖最佳实践，通过三个数据质量层次的渐进式精炼实现数据从原始到可用的转化：

- **Bronze 层**：忠实存储原始数据，不做任何业务转换，仅附加采集元数据（Kafka 偏移量、入库时间等），作为数据的"单一事实来源（Single Source of Truth）"；
- **Silver 层**：在 Bronze 基础上进行清洗、去重与字段标准化，生成经过质量校验的宽表，供下游分析消费；
- **Gold 层**：基于 Silver 层数据进行业务维度聚合（如按话题、按时间粒度的统计指标），直接对接报表与应用层，追求查询效率最优。

本系统严格遵循上述架构思想，各层的具体设计细节将在第 4 章详细阐述。

## 2.4 StructBERT 中文情感分析模型

### 2.4.1 BERT 预训练语言模型基础

BERT（Bidirectional Encoder Representations from Transformers）[Devlin et al., 2019] 是 Google 于 2018 年提出的基于 Transformer 编码器的预训练语言模型，通过掩码语言模型（Masked Language Model，MLM）与下一句预测（Next Sentence Prediction，NSP）两个自监督任务，在大规模无标注语料上习得深层双向语义表示。在下游任务上，BERT 只需追加少量任务特定层并进行微调（Fine-Tuning），即可在情感分析、命名实体识别、自然语言推理等多类任务上取得优异性能。

### 2.4.2 StructBERT 的结构增强

StructBERT [Wang et al., 2020] 是阿里巴巴达摩院在 BERT 基础上提出的结构感知预训练模型，其关键改进在于引入了两个结构增强预训练目标：

- **词序重建目标（Word Structural Objective）**：随机打乱输入序列中若干个词的顺序，要求模型预测原始顺序，迫使模型学习局部语言结构；
- **句子关系预测目标（Sentence Structural Objective）**：不仅预测两个句子是否相邻（NSP），还引入预测句子顺序的任务，增强模型对段落级语义关系的建模能力。

上述结构感知预训练使 StructBERT 在理解中文评论的语义结构（如转折关系、强调结构）时具备更强的表征能力，在 NLPCC、ChnSentiCorp 等中文情感分析数据集上取得了领先性能。

### 2.4.3 本系统中的情感分析设计

本系统采用 ModelScope 开源的 `iic/nlp_structbert_sentiment-classification_chinese-base` 检查点，通过 `modelscope.pipelines` 提供的 `text-classification` Pipeline API 进行调用。在工程实现上，设计了以下关键机制：

**单例模式（Singleton Pattern）**：将模型实例封装为 `StructBERTSentimentService` 类的类变量，通过 `get_instance()` 类方法保证模型在 Driver 进程生命周期内只加载一次，避免每次微批次处理时重复加载模型权重带来的严重性能开销。

**批量推理（Batch Inference）**：将一个微批次内的全部评论文本打包为列表，以 `batch_size=32` 的粒度分组提交 Pipeline，充分利用矩阵运算并行性，相较于逐条推理可获得数倍的吞吐量提升。

**中性类别判定（Neutral Threshold）**：模型原生输出为正向/负向二分类，当最高置信度低于 `STRUCTBERT_NEUTRAL_THRESHOLD`（默认 0.72）时，系统将该条评论的情感标签修正为"中性（neutral）"，以降低低置信度预测对热度统计的干扰。

**异常降级（Fallback Mechanism）**：当批量推理出现异常时，自动退化为逐条推理；若单条推理仍然失败，则以"中性/置信度 0.0"作为兜底结果，保证流处理管道的稳健性不受模型推理异常的破坏。

## 2.5 FastAPI 后端框架

FastAPI 是基于 Python 3.6+ 类型提示（Type Hints）构建的现代高性能 Web 框架，以 Starlette 为 ASGI 服务器基础、以 Pydantic 为数据验证引擎。其核心优势体现在：

- **自动文档生成**：基于函数签名与类型注解，自动生成符合 OpenAPI 3.0 规范的交互式接口文档（Swagger UI / ReDoc），极大地简化了前后端联调成本；
- **异步支持**：原生支持 `async/await` 语法，可充分利用 Python 异步 I/O 能力应对高并发请求；
- **高性能**：在 TechEmpower 基准测试中，FastAPI 的 JSON 序列化与响应吞吐量优于 Flask 等传统同步框架数倍。

本系统的 API 服务层采用 FastAPI 搭建，通过 `deltalake` Python 库直接读取 Delta Lake 表数据并转换为 Pandas DataFrame，经序列化后作为 JSON 响应返回，全程无需启动额外的 Spark 会话，降低了 API 服务的资源消耗。

## 2.6 Vue 3 与 ECharts 可视化技术

### 2.6.1 Vue 3 前端框架

Vue 3 是由尤雨溪（Evan You）主导开发的渐进式 JavaScript 框架，相较于 Vue 2 引入了以下关键改进：

- **Composition API**：以 `setup()` 函数为入口，通过 `ref`、`reactive`、`computed`、`watch` 等组合式函数替代 Options API，使逻辑复用更加灵活；
- **性能优化**：重写了虚拟 DOM 渲染器，引入 Proxy-based 响应式系统，相较于 Vue 2 的 Object.defineProperty 方案具有更完整的响应式覆盖（如数组索引直接赋值）；
- **Tree-Shaking 友好**：模块化设计使打包工具可按需引入，显著减小生产包体积。

本系统前端采用 Vue 3 + Vite 构建工具链，以 `<script setup>` 语法糖为主要编码范式，结合 `<script setup>` 内的 Composition API 实现状态管理与数据获取逻辑的高度内聚。

### 2.6.2 Apache ECharts 数据可视化

ECharts 是由百度开源、现已捐献给 Apache 软件基金会的 JavaScript 数据可视化库，以其丰富的图表类型、流畅的动画效果与高度可定制性著称。本系统使用 ECharts 6.x，在监测大屏中渲染三类趋势图：

- **双系列折线图**：同轴展示评论总数与负面评论数的时间序列，便于对比分析；
- **单系列折线图（负面占比）**：展示以百分比为单位的负面评论占比趋势，辅助识别情绪突变节点；
- **热度折线图**：展示话题热度分数的时间序列，辅助识别事件峰值。

所有图表均集成 DataZoom 组件，支持鼠标滚轮缩放与拖拽选区，提升超长时间序列的交互体验。
